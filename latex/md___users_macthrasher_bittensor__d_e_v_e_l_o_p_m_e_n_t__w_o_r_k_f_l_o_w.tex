Feature branches are used to develop new features for the upcoming or a distant future release. When starting development of a feature, the target release in which this feature will be incorporated may well be unknown at that point.

The essence of a feature branch is that it exists as long as the feature is in development, but will eventually be merged back into {\ttfamily nobunaga} (to definitely add the new feature to the upcoming release) or discarded (in case of a disappointing experiment).\hypertarget{md___users_macthrasher_bittensor__d_e_v_e_l_o_p_m_e_n_t__w_o_r_k_f_l_o_w_autotoc_md57}{}\doxysubsubsection{Release branches}\label{md___users_macthrasher_bittensor__d_e_v_e_l_o_p_m_e_n_t__w_o_r_k_f_l_o_w_autotoc_md57}

Release branches support preparation of a new production release. Furthermore, they allow for minor bug fixes and preparing meta-\/data for a release (e.\+g.\+: version number, configuration, etc.). By doing all of this work on a release branch, the {\ttfamily nobunaga} branch is cleared to receive features for the next big release.

This new branch may exist there for a while, until the release may be rolled out definitely. During that time, bug fixes may be applied in this branch, rather than on the {\ttfamily nobunaga} branch. Adding large new features here is strictly prohibited. They must be merged into {\ttfamily nobunaga}, and therefore, wait for the next big release.\hypertarget{md___users_macthrasher_bittensor__d_e_v_e_l_o_p_m_e_n_t__w_o_r_k_f_l_o_w_autotoc_md58}{}\doxysubsubsection{Hotfix branches}\label{md___users_macthrasher_bittensor__d_e_v_e_l_o_p_m_e_n_t__w_o_r_k_f_l_o_w_autotoc_md58}

Hotfix branches are very much like release branches in that they are also meant to prepare for a new production release, albeit unplanned. They arise from the necessity to act immediately upon an undesired state of a live production version. When a critical bug in a production version must be resolved immediately, a hotfix branch may be branched off from the corresponding tag on the master branch that marks the production version.

The essence is that work of team members, on the {\ttfamily nobunaga} branch, can continue, while another person is preparing a quick production fix.\hypertarget{md___users_macthrasher_bittensor__d_e_v_e_l_o_p_m_e_n_t__w_o_r_k_f_l_o_w_autotoc_md59}{}\doxysubsection{Git operations}\label{md___users_macthrasher_bittensor__d_e_v_e_l_o_p_m_e_n_t__w_o_r_k_f_l_o_w_autotoc_md59}

When finished, the bugfix needs to be merged back into {\ttfamily master}, but also needs to be merged back into {\ttfamily nobunaga}, in order to safeguard that the bugfix is included in the next release as well. This is completely similar to how release branches are finished.

First, update master and tag the release.


\begin{DoxyItemize}
\item Switch branch to master\+: {\ttfamily git checkout master}
\item Merge changes into master\+: {\ttfamily git merge -\/-\/no-\/ff hotfix/3.\+3.\+4/optional-\/descriptive-\/message}
\item Tag new version\+: {\ttfamily git tag -\/a v3.\+3.\+4 -\/m \char`\"{}\+Releasing v3.\+3.\+4\+: some comment about the hotfix\char`\"{}}
\item Pushing changes to master\+: {\ttfamily git push origin master}
\item Pushing tags to origin\+: {\ttfamily git push origin -\/-\/tags}
\end{DoxyItemize}

Next, include the bugfix in {\ttfamily nobunaga}, too\+:


\begin{DoxyItemize}
\item Switch branch to nobunaga\+: {\ttfamily git checkout nobunaga}
\item Merge changes into nobunaga\+: {\ttfamily git merge -\/-\/no-\/ff hotfix/3.\+3.\+4/optional-\/descriptive-\/message}
\item Pushing changes to origin/nobunaga\+: {\ttfamily git push origin nobunaga}
\end{DoxyItemize}

The one exception to the rule here is that, {\bfseries{when a release branch currently exists, the hotfix changes need to be merged into that release branch, instead of}} {\ttfamily nobunaga}. Back-\/merging the bugfix into the {\bfseries{release}} branch will eventually result in the bugfix being merged into {\ttfamily develop} too, when the release branch is finished. (If work in develop immediately requires this bugfix and cannot wait for the release branch to be finished, you may safely merge the bugfix into develop now already as well.)
